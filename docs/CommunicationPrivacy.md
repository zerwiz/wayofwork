# Way of Work: Communication Privacy & Identity Architecture

## Overview
The Way of Work platform uses a secure communication architecture for its AI-powered messaging bots (Telegram, WhatsApp, Email). This document outlines how user identity is verified, how data isolation (multi-tenancy) is enforced, and how privacy is maintained.

## 1. User Identity Resolution
When a message is received from an external channel (e.g., Telegram), the system must map the anonymous channel ID to a known, authorized user in the Way of Work platform.

### Identity Mapping
The platform uses the `user_channel_links` table to store this association:
*   **Key Mapping**: `channel_user_id` (e.g., Telegram ID) ↔ `user_id` + `tenant_id`.
*   **Verification**: Upon receiving a message, the `channel-router.ts` queries this table.
    *   If no entry exists, the system forces the user to link their account via the web portal.
    *   If an entry exists, the system identifies the `user_id`, `role`, and `tenant_id` for the conversation.

## 2. Data Isolation & Privacy (Multi-Tenancy)
The system enforces strict data privacy through two primary layers:

### A. Database-Level Isolation
All internal tool calls and database queries executed by the AI agent are scoped to the authenticated `tenant_id`.
*   **Technical Implementation**: Queries generated by the orchestrator are automatically appended with `AND tenant_id = ?`.
*   **Scope**: It is technically impossible for an agent to query data belonging to a different organization (tenant).

### B. Agent System Prompt (Privacy Rules)
Even if a user *has* access to the system, the agent is restricted by its system prompt (found in `.wo/agents/`).

The agent is explicitly instructed with the following **Privacy Rules**:
1.  **Strict Isolation**: The agent only has context for the current conversation. It does not have access to messages or data belonging to other users.
2.  **No Leaking**: The agent is forbidden from revealing time logs, tasks, or personal information belonging to other users.
3.  **Authentication Guard**: If a user attempts to query information about a colleague, the agent is trained to respond: *"I am sorry, I can only provide information for your own account."*

## 3. Human-in-the-Loop (HITL)
To prevent unauthorized data manipulation:
*   **No Direct Writes**: AI agents have **zero** permissions to write directly to production tables.
*   **Approval Queue**: All requests to change data (e.g., logging time, assigning tasks) are routed through the `pending_changes` system.
*   **Manual Review**: These proposals must be approved by an Admin in the Admin Console before being committed to the database.
